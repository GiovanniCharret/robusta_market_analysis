Arquitetura "Static-first" para o site dos destaques do ROBUSTA

Site privado de uso pessoal, hospedado em VPS Linux com domínio próprio, mostrando destaques da última execução e drill-down por ticker. Pipeline Python (já existente em robusta/) roda 3× ao dia via cron, grava JSONs em disco, e o nginx serve esses JSONs como arquivos estáticos para um frontend de HTML + JavaScript vanilla.

01

Diagrama da arquitetura

Três processos no VPS: o cron dispara o pipeline nos horários fixos; o pipeline Python baixa dados das fontes externas e escreve JSONs no disco; o nginx serve esses JSONs e os arquivos do frontend para qualquer browser que abrir o domínio. Não há aplicação Python rodando enquanto o usuário navega — só arquivos.

scheduler (cron) processo Python (executa e morre) arquivo em disco servidor sempre ligado (nginx) fonte externa

O pipeline é um processo curto: dispara, baixa, grava, termina. Entre execuções o VPS só tem o nginx ativo, servindo arquivos. Nenhum processo Python permanente — sem uvicorn, sem worker, sem fila.

02

Layout em disco no VPS

Três áreas separadas, seguindo a convenção FHS do Linux: código em /opt, estado mutável em /var/lib, conteúdo servido pelo nginx em /var/www. Cada uma tem dono e permissões próprios, e pode ser feita backup ou redeployada independente. 

/opt/robusta/                  # código (git clone do repo)
├── .venv/                  # venv Python isolado
├── robusta/                # o pacote Python (já existe)
├── scripts/
│   └── run_pipeline.sh     # wrapper chamado pelo cron
├── lista_tickers_liquidos.xlsx
└── all_ticker_financial_indicators.xlsx

/var/lib/robusta/runs/        # estado: JSONs gerados a cada run
├── latest.json             # cópia da run mais recente (dashboard)
├── 2026-05-25T12-30-00Z.json  # arquivo bruto da execução
├── 2026-05-25T16-00-00Z.json
├── 2026-05-25T21-30-00Z.json
└── tickers/                # JSON por ticker (drill-down)
    ├── PRIO3.json           # última foto do PRIO3
    ├── ASAI3.json
    └── ...

/var/www/robusta/             # frontend estático (servido pelo nginx)
├── index.html              # dashboard
├── ticker.html             # página de drill-down
└── assets/
    ├── app.js              # vanilla JS, ~200 LoC
    └── app.css

/etc/cron.d/robusta           # 3 linhas, uma por horário
/etc/nginx/sites-available/robusta  # server block
Por que separar /opt, /var/lib e /var/www?
Permite redeployar o código (/opt) sem tocar nos JSONs históricos (/var/lib), e atualizar o frontend (/var/www) sem re-rodar o pipeline. Backup do estado é um tar só de /var/lib/robusta/.
03

Decisões-chave

As decisões abaixo definem o caráter da arquitetura. Cada uma é reversível, mas mudá-las implica revisitar várias seções do design.

Decisão
Escolha
Por quê
Camada de serviço
nginx servindo estático
Sem histórico/filtros server-side, FastAPI vira overhead. JSON + fetch() resolve. Migrar para FastAPI depois é trivial (mesmos JSONs, novo endpoint que os lê).
Persistência
Arquivos JSON em disco
Alinhado à Fase 6 do PLAN.md. Sem banco de dados, sem migrations. Cada run é um arquivo imutável; latest.json é sobrescrito atomicamente (write + rename).
Frontend stack
HTML + vanilla JS
Dashboard de 1 página + página de ticker — não justifica React/Vue/build step. Sem npm, sem bundler, sem Node no VPS. Carrega em < 50 KB.
Scheduler
cron do sistema
Já existe, é confiável, logs vão para syslog. Alternativa systemd timer tem features que não precisamos. O comando schedule da Fase 8 do PLAN vira opcional — cron cobre.
HTTPS
Let's Encrypt via certbot
Gratuito, renovação automática, padrão de mercado. Configurado uma vez no setup do VPS.
Deploy de código
git pull + systemctl reload nginx
Sem CI/CD para começar. Você empurra para o GitHub, faz SSH no VPS, roda git pull. Quando virar incômodo, vira um GitHub Action de 10 linhas.
04

Fluxo de uma execução (13:00 BRT, terça-feira)

Sequência cronológica de uma run típica, do gatilho do cron até o JSON aparecer no browser.

T+0s

cron dispara run_pipeline.sh

Wrapper ativa o venv, exporta variáveis de ambiente e chama python -m robusta run --emit-json /var/lib/robusta/runs/. Saída redirecionada para /var/log/robusta/run.log.

T+0s

Pipeline lê lista_tickers_liquidos.xlsx

~369 tickers carregados. Cache de fundamentos (all_ticker_financial_indicators.xlsx) já está no repo; só re-raspa no 1º dia útil do mês ou com --refresh-fundamentos.

T+1m

Análise técnica (Yahoo Finance, 1 req/ticker)

screener() baixa 5 anos de OHLCV por ticker e calcula MMAs, volatilidade, volume persistente, níveis. tqdm loga progresso no run.log.

T+6m

Análise fundamentalista (cache, sem rede)

adicione_indicadores_e_ranking() recalcula P/L, P/VP, Dív.Líq./VM com preços recém-baixados; ranqueia decis; gera score.

T+6m

Merge + ranking cross-sectional

executa_pipeline() devolve um RunResult (técnica + fundamental + signals).

T+6m

Persistência (Fase 6, novo módulo)

robusta/persistence.py serializa o RunResult para JSON. Escreve runs/2026-05-25T16-00-00Z.json, quebra em runs/tickers/<ticker>.json, e faz rename atômico para latest.json.

T+6m

Browser do usuário

Próxima vez que você abrir seu-dominio.com, o JS faz fetch('/data/latest.json') e popula o dashboard. Você clica em PRIO3 → fetch('/data/tickers/PRIO3.json') abre o drill-down.

Tratamento de falhas (a detalhar na seção 5 do brainstorm)
Se o pipeline falhar no meio, latest.json continua válido (a run anterior). Os JSONs novos só são "comitados" no fim, via rename atômico. Falhas individuais de ticker já são logadas em warnings/failed_tickers dentro do próprio RunResult.

06

Perguntas em aberto desta seção

Antes de avançar para a seção 2/6 (contrato JSON), preciso da sua resposta a três coisas:

1. Os caminhos /opt/robusta, /var/lib/robusta, /var/www/robusta te servem?
Opção A. Seguir convenção FHS (recomendado): cada coisa em seu lugar canônico, permissões separadas. Opção B. Tudo dentro de /home/seu_usuario/robusta/ para simplicidade (não precisa sudo para mexer).
2. Roda só dias úteis ou todo dia?
Opção A. Só seg–sex (recomendado): mercado fechado no fim de semana, dados não mudam, evita gastar quota de API à toa. Opção B. Todo dia: você quer abrir o site sábado e ver alguma coisa atualizada (mesmo que igual à de sexta).
3. Domínio
Opção A. Domínio próprio que você já tem (ex.: robusta.seudominio.com) — você cuida do DNS. Opção B. Vai comprar um (sugiro Cloudflare Registrar ou Registro.br para .com.br). Opção C. Só IP do VPS por enquanto, sem domínio (sem HTTPS válido — funciona, mas browser xinga).
